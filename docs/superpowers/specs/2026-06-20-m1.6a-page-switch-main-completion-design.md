# M1.6a ‚Äî Page-switch infrastructure + Main page completion

**Status:** Design approved 2026-06-20. First sub-task of M1.6 (remaining editor tabs).
**Depends on:** M1.5 (OSC + FILTER vertical slice, gate passed 2026-06-20).
**Authoritative requirements:** [`docs/spec/editor-requirements.md`](../../spec/editor-requirements.md) ¬ßM1.6; [`ROADMAP.md`](../../../ROADMAP.md) ¬ßM1.6.

## Context ‚Äî what M1.5 actually shipped

`PageComponent::buildWidgets` (`source/mw2xtEditor/PageComponent.cpp`) iterates **every**
child of the skin page node and wires any node whose `parameterAttachment.parameter`
resolves in `ParamRegistry`. The skin's `pageOsc` node is the whole **Main sound page** ‚Äî
84 parameter attachments spanning OSC, Wave-mix, AMP, Amp-Env, FILTER, Filter-Env, LFO1
and LFO2. Because all of those params resolve in the registry, **M1.5 already built and
wired every one of those sections.** "OSC + FILTER" named the *gate-test focus*, not a
limit on what was rendered.

Consequences for the M1.6 split (decided 2026-06-20: **follow the skin**):

- The roadmap "tabs" AMP / ENV / LFO / MISC / WAVE-select are **sections on the Main
  page**, already on screen, not separate tabs. This matches the cross-tab requirement
  "all section parameters visible without page navigation (mirrors hardware section
  grouping, not the page system)."
- The dedicated `pageWave` skin page is the graphical **wave editor** ‚Üí Phase 3 (M3),
  not M1.6.
- The `pageMod` page is the **mod matrix** ‚Üí Phase 4 (M4), not M1.6.
- Later M1.6 sub-tasks (b‚Äìh) are mostly **graphical affordances layered over the
  finished Main page** plus the genuinely separate ARP / MULTI pages and custom
  NAME / GLOBAL panels.

## Scope of M1.6a

M1.6a delivers the capability every later sub-task depends on, plus the clean
completion of the Main page:

1. **Page-switch infrastructure** ‚Äî `PageContainer` + radio-grouped page buttons.
   *Primary deliverable.*
2. **Conditional image overlays** ‚Äî the one remaining Main-page widget type with a
   clean, well-specified skin contract (the `F1Type` filter-type icon).
3. **Main-page bidirectional-sync audit** ‚Äî confirm AMP / ENV / LFO / MISC sync both
   directions (built in M1.5, never formally verified); fix any mis-ranged param or
   empty-items combo found.

The `wtMinus` / `wtPlus` Wave inc/dec buttons are **deferred to M1.6d** ‚Äî the skin nodes
carry no `parameterAttachment`, so their target param is undeclared and must be mapped
explicitly alongside the WAVE preview work.

## Architecture

Approach chosen: **page container + per-tab Components, lazily instantiated, switched by
the skin's existing page buttons.** Generic tabs reuse the unchanged M1.5 `PageComponent`;
bespoke tabs (built in later sub-tasks) are custom Components; visibility switching is
`setVisible`, never rebuild or re-layout ‚Äî preserving the M1.5 repaint-storm fix.

### Component 1 ‚Äî `PageContainer`

A `juce::Component` that owns the pages and shows exactly one at a time.

```cpp
enum class PageId { Main, Mod, Arp, Multi, Wave, PatchManager };

class PageContainer : public juce::Component {
public:
    // page may be nullptr for a not-yet-built slot.
    void   addPage(PageId id, std::unique_ptr<juce::Component> page);
    void   showPage(PageId id);        // hides others; sets visibility only
    PageId currentPage() const noexcept;

    std::function<void(PageId)> onPageChanged;   // drives the button bar

    void resized() override;           // forwards full bounds to every page
};
```

- **Lazy instantiation:** `addPage` accepts `nullptr`; `showPage` on an empty slot shows
  a background-only placeholder `PageComponent` (skin ships `xtPageMod/Arp/Multi/Wave.png`,
  so placeholders render real backgrounds).
- **Shared bounds:** every page fills the page area `EditorComponent::resized` already
  computes for `pageOsc`. Switching is `setVisible(true/false)` ‚Äî no re-layout, no rebuild.
- **Persistent instances:** instantiated pages stay alive when hidden, so each page's
  model `ChangeListener` keeps its widgets in sync; switching back shows current state.

**Invariant:** exactly one page is visible at any time.

### Component 2 ‚Äî Page button bar

The skin defines six toggle buttons (`pageButtonMain/Mod/Arp/Multi/Wave/PatchManager`)
at baked coordinates in the top bar, each with on/off frames in the `xtknob` 2-frame
sheet.

- Build them as `SpritesheetButton`s in a **radio group**: clicking one sets it on,
  clears the others, and calls `container.showPage(id)`.
- `PageContainer::onPageChanged` drives the reverse direction so button state stays
  correct if a switch is triggered programmatically.
- M1.6a mapping: **Main** ‚Üí real page; **Mod / Arp / Multi / Wave** ‚Üí placeholders;
  **PatchManager** ‚Üí wired but disabled (greyed), since it is M2. This exercises radio
  switching across the real buttons without faking the buttons themselves.

### Component 3 ‚Äî `ConditionalImage` widget

Handles skin nodes that pair `image` with `condition`
(`f1TypeUnused`, `f1Type6/7/8/9/12`):

```cpp
// skin contract, verified in xtDefault.json:
//   "condition": { "enableOnParameter": "F1Type", "enableOnValues": "6" }
class ConditionalImage : public juce::Component {
public:
    ConditionalImage(juce::Image img, int sdataIndex, std::vector<int> enableValues);
    void refreshFromModel(uint8_t rawValue);   // setVisible(value ‚àà enableValues)
    void paint(juce::Graphics&) override;       // draws img when visible
};
```

- `PageComponent::buildWidgets` gains a branch: a child with `image` + `condition`
  resolves `condition.enableOnParameter` via `ParamRegistry` ‚Üí SDATA index, parses
  `enableOnValues` (space-separated) ‚Üí `enableValues`, creates the widget, and refreshes
  it in `refreshFromModel`.
- Pure indicators: **no** input, **no** undo, **no** SNDP. They render the filter-type
  graphic over the Main page when `F1Type` matches ‚Äî completing the FILTER visual M1.5
  left as a stub.

### EditorComponent integration

- Replace the single `std::unique_ptr<PageComponent> pageOsc` with a `PageContainer`
  plus the six page buttons.
- `resized()` keeps the existing page-area bounds; the container fills that area and
  forwards bounds to the visible page.
- The hand-built top-bar `TextButton`s (Connect / Req Patch / Undo / Redo / A‚ÜîB /
  Push Patch) are untouched; the skin page buttons are a separate baked group.

## Main-page sync audit

No new wiring is expected. M1.6a formally verifies, against Xenia, that AMP, ENV
(amp-env + filter-env knobs), LFO1 / LFO2, and MISC (Aliasing / TimeQuant / Accuracy /
Clipping) each:

- push SNDP on a UI edit (UI ‚Üí model ‚Üí hardware), and
- update the widget on inbound SNDD / CC (hardware ‚Üí model ‚Üí UI).

Any param found mis-ranged, or any combo with empty `items` (e.g. an LFO-shape selector),
is fixed here. This audit produces the concrete output the roadmap asks for: **real
per-tab sizing learned from one finished page**, before estimating sub-tasks b‚Äìh.

## Testing & acceptance

**Unit-testable (ctest), JUCE-light with a fake/headless model:**

- `ConditionalImage` visibility logic: visible iff value ‚àà `enableValues`.
- `PageContainer::showPage` invariant: exactly one page visible after any switch.
- Page-button radio-group exclusivity: selecting one clears the others.

**Manual / Xenia:**

- Switch Main ‚Üî placeholder pages with no visual glitch mid-SysEx (M1.5 NFR carries over).
- `F1Type` filter icon changes as the filter-type combo changes.
- AMP / ENV / LFO / MISC parameters round-trip on Xenia (and spot-check on real XT).

**Exit criteria (roadmap ¬ßM1.6, per-tab):**

- [ ] All Main-section params visible without page navigation (already true; confirmed).
- [ ] Bidirectional sync (UI ‚Üî XT) verified for every Main-page control.
- [ ] First tab-specific visual affordance built: page switching + the `F1Type` conditional
      icon.

## Out of scope / deferred / open questions

- **`wtMinus` / `wtPlus`** ‚Üí M1.6d (Wave inc/dec; no declared target param).
- **Graphical affordances** (graphical ADSR, 8-stage wave-env, LFO waveform icons,
  inline waveform preview) ‚Üí M1.6b/c/d as overlays on the finished Main page.
- **NAME & GLOBAL navigation ‚Äî open question for M1.6g/h.** The skin has no page button
  for either. Options: add custom buttons, surface NAME as a Main-page field, or place
  GLOBAL behind a settings affordance. Flagged, not solved in M1.6a.
- **MOD (M4), Wave editor (M3), PatchManager (M2)** pages remain placeholders, reachable
  via their buttons (PatchManager disabled).
